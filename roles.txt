SEMKO ROLE AND PERMISSION REFERENCE
===================================

Purpose
-------
This file gives you a clean, backend-aligned reference for creating roles in Django admin.
All permission codes below are taken from the backend constants and are ready to paste into
the `permissions` list field on each Role record.

Core Rule
---------
1. Use Django `is_superuser=True` only for true system ownership.
2. Use Role permissions for all normal business users.
3. Prefer Role-based access over direct per-user permissions.
4. The frontend sidebar is permission-driven, so users will only see modules that match
   the permissions in their role or effective permissions.

ROLE DESIGN PRINCIPLES
----------------------
System Administrator
- Technical owner
- Best handled with Django superuser
- Should see everything automatically

Director
- Broad visibility across the app
- Prefer Role-based full access instead of Django superuser
- Good for executives who need app-wide visibility but not technical system root access

Department Roles
- Narrow roles should only see their working modules
- Example: Trips Officer should see Trips and supporting master data, not Payroll or Users

WHAT TO DO FOR YOUR OWN ACCOUNT
-------------------------------
For your own admin account:
- keep `is_superuser=True`
- keep `is_staff=True`
- role can be left blank, or assigned to a full-access role if you want consistency

Important:
- an empty role with `permissions=[]` gives a normal user almost nothing
- a superuser should still see everything because the frontend now honors `is_superuser`


WHAT TO DO FOR A DIRECTOR USER
------------------------------
Best practice:
- create a normal user
- set `is_staff=False` unless they must access Django admin
- assign the `Director` role
- do NOT make them Django superuser unless they truly need system-root powers

If the Director must use Django admin too:
- set `is_staff=True`
- keep them non-superuser unless truly necessary
